# PNF-Registration Feature Analysis (sdnr-lighty)

Based on the exploration of the `sdnr-lighty` repository, I have analyzed the PNF-Registration (Physical Network Function) support added to PantheonTech Lighty to make it act as an SDNR for O-RAN. 

Here is a comprehensive breakdown of the code and the architectural changes made by your former colleague.

## 1. High-Level Architecture
The PNF Registration feature allows the SDNR to automatically discover and mount new PNF devices (like O-RU/O-DU) by listening to **VES (VNF Event Stream)** messages over a Kafka message bus. When a PNF registration event is received, the application automatically provisions a NETCONF mount point for that device within the SDN-R controller.

## 2. Relationship to the Pantheon Lighty Repository
The `sdnr-lighty` repository is a downstream application built **on top of** the core framework provided by the `lighty` repository.

1. **The Framework (`lighty` repository)**: This is the official PANTHEON.tech Lighty.io codebase. It provides the core engine (`lighty-core`) and modules (`lighty-netconf-sb`, `lighty-restconf-nb-community`, etc.) that allow developers to run OpenDaylight (ODL) components as a standard Java application, without the overhead of an OSGi container (like Apache Karaf).
2. **The Custom Application (`sdnr-lighty` repository)**: This is a custom deployment/application created by your team. It consumes the artifacts built by the `lighty` repo to spin up a specialized SDN-R instance. Your colleague took the Lighty base and bolted on the `PnfModule` to create a tailored SDN-R product.

## 3. Project Structure
The code is split into two main areas:
- **`modules/iosmcn-pnf-registration`**: Contains the core business logic, Kafka consumers, and REST clients for PNF Registration.
- **`applications/iosmcn-pnf-registration-aggregator`**: Contains the application launcher that integrates the `PnfModule` into the Lighty.io controller lifecycle.

## 4. Key Components and Code Flow

### A. The Application Entry Point
[Main.java](file:///c:/Users/sridh/Documents/GitHub/sdnr-lighty/applications/iosmcn-pnf-registration-aggregator/iosmcn-pnf-registration-app/src/main/java/org/iosmcn/lighty/applications/pnfreg/app/Main.java)

The `Main` class initializes the entire Lighty.io ecosystem. Your colleague modified this file to include the new `PnfModule`.
- It starts the core Lighty Controller, RESTCONF server, and NETCONF Southbound plugin.
- It also starts the Netconf CallHome plugin.
- Finally, it instantiates and starts the `PnfModule` (around line 252), integrating it as part of the controller's startup sequence.

### B. The PNF Module Lifecycle
[PnfModule.java](file:///c:/Users/sridh/Documents/GitHub/sdnr-lighty/modules/iosmcn-pnf-registration/src/main/java/io/lighty/modules/pnf/registration/PnfModule.java)

This class implements `AbstractLightyModule` and manages the lifecycle of the PNF feature.
- **Initialization**: During `initProcedure()`, it reads the configuration file `config/mountpoint-registrar.properties`.
- **Kafka Consumers**: If Strimzi Kafka is enabled in the configuration, it launches `StrimziKafkaVESMsgConsumerMain` in a new background thread. This thread sets up the various Kafka consumers for different VES domains (PNF Registration, Faults, etc.).
- It also implements an `IConfigChangedListener` to dynamically start or stop the Kafka consumers if the configuration changes at runtime.

### C. Consuming PNF Registration Events
[StrimziKafkaPNFRegVESMsgConsumer.java](file:///c:/Users/sridh/Documents/GitHub/sdnr-lighty/modules/iosmcn-pnf-registration/src/main/java/io/lighty/modules/pnf/registration/mountpointregistrar/vesdomain/pnfreg/StrimziKafkaPNFRegVESMsgConsumer.java)

This is the core class responsible for handling incoming PNF registration VES messages from Kafka.
- **Message Parsing**: When `processMsg(String msg)` is invoked, it parses the JSON event. It ignores messages generated by the SDNR itself to avoid loops.
- **Field Extraction**: It extracts crucial device details such as `pnfId`, IP Address, Protocol (`SSH` or `TLS`), Port (defaults to `17830`), and credentials (`username`, `password`, or TLS `keyId`).
- **Validation**: It checks if any required fields are missing.
- **Action**: Once the data is parsed and validated, it bundles these parameters into a payload map and passes it to the `PNFMountPointClient` to actually mount the device.

### D. Creating the NETCONF Mount Point
[PNFMountPointClient.java](file:///c:/Users/sridh/Documents/GitHub/sdnr-lighty/modules/iosmcn-pnf-registration/src/main/java/io/lighty/modules/pnf/registration/mountpointregistrar/vesdomain/pnfreg/PNFMountPointClient.java)

This REST client connects back to the SDN-R controller's own RESTCONF API to create the mount point.
- **Payload Templates**: It contains XML templates for both `SSH_PAYLOAD` and `TLS_PAYLOAD`. These templates define the Netconf Node structure expected by the `network-topology` model.
- **Preparation**: The `prepareMessageFromPayloadMap` method takes the extracted PNF data and substitutes the placeholders (like `@device-ip@`, `@device-port@`, etc.) in the XML templates with the real values.
- **Execution**: The `sendNotification` method executes an HTTP PUT request to the RESTCONF URL: `rests/data/network-topology:network-topology/topology=topology-netconf/node=<pnfId>`, which tells the OpenDaylight/Lighty controller to initiate a NETCONF connection to the PNF device.

## Summary of the Flow
1. The Lighty **Main** application starts and initializes the **PnfModule**.
2. **PnfModule** reads configs and starts the **Kafka Consumer thread**.
3. A new PNF device comes online and publishes a VES PNF Registration event to the Kafka topic.
4. **StrimziKafkaPNFRegVESMsgConsumer** receives the message, parses the device's IP, port, and credentials.
5. **PNFMountPointClient** formulates an XML payload and sends a PUT request to the SDNR's RESTCONF API to create a Netconf mount point for the newly discovered PNF.
